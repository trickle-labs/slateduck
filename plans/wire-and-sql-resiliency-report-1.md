# SlateDuck Resiliency Report: Hardening SQL Classification & PgWire Compatibility

Date: 2026-05-27  
Status: Proposed  
Scope: Architecture, SQL Parser (`slateduck-sql`), and PgWire Server (`slateduck-pgwire`)

---

## 1. Executive Summary

SlateDuck functions as a serverless catalog sidecar, bridging the DuckDB client with the SlateDB key-value storage substrate. To do this, it employs a custom **SQL Statement Classifier** and a **PgWire protocol implementation**. 

While this architecture is highly performant and lightweight, it introduces a **high structural coupling** to the specific query patterns, subquery structures, and initialization sequences generated by DuckDB's native extensions (`postgres_scanner` and `ducklake`). Any change in client-side SQL dialect translation, query projection wrapping, connection pool initialization, or PostgreSQL catalog queries (e.g., in a minor DuckDB update) runs a severe risk of breaking SlateDuck's pattern matching—resulting in connection hangs or `Unsupported` sqlstate errors.

This report outlines **five concrete, production-grade resiliency mitigations** SlateDuck can implement to completely decouple its AST matching from specific client dialects, gracefully absorb handshake changes, and guarantee long-term stability against upgrades in external analytical clients.

---

## 2. The Core Resiliency Vulnerabilities

Our deep assessment of the SQL classifier and PgWire executor reveals three critical points of fragility:

```
┌───────────────────────────┐     ┌────────────────────────────────────────────────────────┐
│   DuckDB Client Upgrade   │ ──> │ Modifies session setups, wraps catalog reads in CTEs,  │
│                           │     │ or adds custom projection subqueries.                  │
└───────────────────────────┘     └──────────────────────────────┬─────────────────────────┘
                                                                 │
                                                                 ▼
┌───────────────────────────┐     ┌────────────────────────────────────────────────────────┐
│  SlateDuck Bounded SQL    │ <── │ Custom AST pattern matcher fails to recognize query;    │
│  Classifier & AST Matcher │     │ falls back to "Unsupported" -> connection breaks.      │
└───────────────────────────┘     └────────────────────────────────────────────────────────┘
```

1. **AST Pattern Rigidity**: The `classify_ast` and `classify_query` functions match precise AST node layouts (e.g. `SetExpr::Select(inner_select)` expecting standard derived tables). A minor compiler change that shifts table-alias scopes or introduces nested query expressions breaks the classifier.
2. **Hardcoded Session Configuration Queries**: Queries like `SELECT pg_database_size(current_database())` or RDS-checks are handled via hardcoded AST checks. If the client expands this list or structures them differently, the parser throws an error.
3. **Implicit Network Blocking in Tests**: Integration tests that validate compatibility with live DuckDB binaries (such as `v0276_lifecycle_tests.rs`) spawn external commands without boundaries, exposing test suites to infinite hangs if network resources are constrained or slow.

## 3. Concrete Mitigations & Architecture Roadmaps

To resolve these vulnerabilities, we propose the following five architectural roadmaps.

---

### Mitigation 1: Abstract Virtual SQL Query Engine (DataFusion Integration)

Currently, when a client issues a `SELECT` query against one of the 28 DuckLake catalog tables, SlateDuck classifies the AST to match specific predefined query shapes (e.g., `SelectMaxSnapshot`, `SelectDataFiles`). This requires SlateDuck to know exactly *how* DuckDB writes its metadata queries.

```
[CURRENT PATH]
Client SQL ──> sqlparser-rs ──> Custom AST Pattern Matcher ──> Hardcoded Handler

[PROPOSED PATH]
Client SQL ──> DataFusion Query Planner ──> Virtual Catalog Tables (Mem-backed) ──> Response
```

#### Proposed Change
Instead of hardcoding handlers for every possible client-side query pattern, SlateDuck should leverage the **`slateduck-datafusion` catalog provider** inside the PgWire executor.
* **Mechanism**: Register the 28 DuckLake catalog tables as memory-backed logical schemas within an in-memory DataFusion `SessionContext` at the start of a PgWire connection.
* **Impact**: When a `SELECT` statement is received from the client, pass it directly to DataFusion to plan and execute against the virtual tables. 
* **Benefits**: 
  - **100% Query Resilience**: SlateDuck automatically gains support for *any* SELECT query structure, including complex subqueries, Common Table Expressions (CTEs), custom projections, Joins, and aggregations, without needing a single change to the `slateduck-sql` AST parser.
  - **Future-Proofing**: If DuckDB's analytical engine changes how it fetches catalog rows, SlateDuck absorbs the query naturally through standard SQL planning.

---

### Mitigation 2: AST Normalizer & Pre-Processing Pipeline

If compiling a full SQL execution engine inside the PgWire executor is too heavy for lightweight sidecar deployments, SlateDuck should insert an **AST Normalization Layer** before the statement classifier.

```
Raw SQL ──> sqlparser-rs AST ──> AST Normalizer ──> AST Pattern Classifier ──> Handler
                                 (Unwraps projections,
                                  strips schemas,
                                  flattens subqueries)
```

#### Proposed Change
Create an AST visitor pipeline (`crates/slateduck-sql/src/classifier/normalize.rs`) that runs before `classify_ast`. The normalizer will:
1. **Recursively Flatten Subqueries**: Collapse nested `TableFactor::Derived` subqueries and lift projection aliases.
2. **Normalize Identifiers**: Uniformly strip catalog schema prefixes (e.g., mapping both `"public".ducklake_table` and `public.ducklake_table` to a canonical `ducklake_table` string).
3. **Format Standardizer**: Strip redundant parentheses, double-quotes, and whitespace tokens.
4. **Remove Unused AST Clauses**: Strip out `LIMIT` or `ORDER BY` clauses that do not impact catalog query routing before matching the core relation.

---

### Mitigation 3: Dynamic Session Settings Registry

DuckDB regularly issues connection setups like `SET client_encoding = 'UTF8'` or `SET DateStyle = 'ISO'`. SlateDuck currently checks these queries via specific string pattern matching in `mod.rs`.

#### Proposed Change
Implement a generic, session-scoped key-value settings map in the PgWire `SessionState` struct (`crates/slateduck-pgwire/src/session.rs`):
```rust
pub struct SessionState {
    settings: HashMap<String, String>,
    // existing state...
}
```
* **Mechanism**: Update `classify_statement` to parse any `SET <variable> = <value>` dynamically as a generic `StatementKind::SetVariable(key, value)`. The PgWire executor will record this pair in `SessionState` and immediately return a standard PostgreSQL `CommandComplete` tag of `"SET"`.
* **Impact**: Decouples SlateDuck from having to hardcode support for every individual configuration variable issued by different client drivers.

---

### Mitigation 4: Automated Dialect Fuzz Testing & SQLSTATE Hardening

To guarantee that unhandled SQL queries never result in server crashes, connection hangs, or uninformative protocol responses, SlateDuck should implement structured dialect fuzzing.

#### Proposed Change
1. **Fuzz Pipeline**: Create a CI target (`tests/dialect_fuzz.rs`) that generates semi-randomized PostgreSQL-dialect query strings and sends them to the PgWire executor.
2. **SQLSTATE 0A000 Coverage**: Ensure that the `execute_sql` handler handles all unsupported queries by returning a clean, standardized PostgreSQL error structure:
   - **SQLSTATE**: `0A000` (Feature Not Supported)
   - **Severity**: `ERROR`
   - **Message**: "Statement is not supported by SlateDuck's catalog facade."
3. **Assert Non-Blocking**: Verify that under fuzz testing, the server never drops the connection abruptly, panics, or hangs, always returning a proper SQLSTATE response to keep the client session alive.

---

### Mitigation 5: Hardened Testing with Sandbox Timeouts

As highlighted during our test suite assessment, integration tests that call external command processes can block indefinitely if client dependencies (like the `ducklake` extension) attempt to connect to external endpoints or fetch updates over restricted networks.

#### Proposed Change
Harden `crates/slateduck-pgwire/tests/v0276_lifecycle_tests.rs` and other process-spawning targets by implementing strict tokio-timeouts around external commands:

```rust
// Replace blocking `Command::output()` calls in helpers:
let output = tokio::time::timeout(
    Duration::from_secs(5),
    tokio::process::Command::new("duckdb")
        .arg("-c")
        .arg("LOAD ducklake; SELECT 1;")
        .output()
)
.await;

let available = match output {
    Ok(Ok(o)) => o.status.success(),
    _ => false, // Times out after 5s and skips cleanly without hanging the suite
};
```

## 4. Implementation Priority & Severity Assessment

We recommend implementing these mitigations in the following prioritized sequence:

| Priority | Mitigation Strategy | Impact | Effort | Target Crate |
| :---: | :--- | :--- | :---: | :--- |
| **P0** | **Hardened Testing Sandbox Timeouts** | Stops test execution suite hangs in network-restricted local or CI environments. | Low | `slateduck-pgwire` (Tests) |
| **P1** | **Dynamic Session Settings Registry** | Gracefully handles session-setup commands from different PostgreSQL drivers. | Low | `slateduck-pgwire` / `slateduck-sql` |
| **P1** | **AST Normalization Layer** | Reduces classification fragility by rewriting complex subqueries before matching. | Medium | `slateduck-sql` |
| **P2** | **Abstract Virtual SQL Engine** | Completely eliminates classifier vulnerability by executing arbitrary catalog SELECTs via DataFusion. | High | `slateduck-datafusion` / `slateduck-pgwire` |
| **P2** | **Dialect Fuzzing & SQLSTATE Conformance** | Guarantees connection stability and graceful error fallbacks under arbitrary queries. | Medium | `slateduck-pgwire` (Tests) |

---

## 5. Summary of Recommended Path

By executing these mitigations, SlateDuck will transition from a catalog tool that relies on fragile, hardcoded query matching to a **fully resilient PostgreSQL-wire protocol sidecar**. Hardening the test sandbox with timeouts addresses immediate CI reliability risks, while abstracting query execution via a virtual DataFusion session provides long-term, structural immunity to changes in analytical client SQL dialects.
